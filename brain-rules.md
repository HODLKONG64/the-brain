## RULE CONFLICT GATE — RCG (locked from all chats)
This gate is the safety mechanism that prevents any new or updated rule from breaking the infinite lore flow. It runs EVERY time a rule is applied, updated, or when the agent encounters something that suggests a rule may need revision.

### RCG-1 — Triple-Check Protocol
Before any rule is applied in a lore or art generation pass, the agent must internally verify:
CHECK 1 — Accuracy: Is this rule still accurate based on the latest official online data? If a crawl in the last 2 hours found conflicting official data, the rule must be updated first.
CHECK 2 — Consistency: Does this rule produce an output consistent with the last 7 days of lore history? If applying the rule would create a jarring continuity break, the rule is softened to a strong suggestion for this post only, and a lore-coherent bridge sentence is added.
CHECK 3 — Flow: Does applying this rule make the lore feel organic and alive, or does it make it feel mechanical and forced? If mechanical, the agent must find the most natural way to honour the rule's intent while maintaining the mind-log voice.

### RCG-2 — Rule Staleness Flag
If a rule has not produced a positive lore outcome in the last 20 posts (tracked in bot-state.json as `rule_last_positive`), the agent flags it as "stale" and reduces its weighting to 30% for the next 10 posts. If it still underperforms, it logs a "RULE REVIEW NEEDED" note in lore-history.md for the user to inspect. The rule is NOT deleted automatically — only a human can permanently remove a rule.

### RCG-3 — Conflict Severity Scale
When a conflict is detected, the agent rates it on a 3-level scale and responds accordingly:
- **LEVEL 1 — Minor inconsistency**: Small detail mismatch (e.g. character wearing wrong hat in one scene). Agent fixes silently, logs in lore-history.md, continues.
- **LEVEL 2 — Moderate continuity break**: Would require noticeable retconning. Agent adds a brief in-universe bridge line explaining the change, logs the fix, continues.
- **LEVEL 3 — Major lore break**: Would fundamentally contradict an active arc or established canon fact. Agent does NOT apply the conflicting element. It generates the post without that element, logs a "RCG LEVEL-3 BLOCK — [description]" entry in lore-history.md, and flags it for user review.

### RCG-4 — Rule Update Protocol
When the agent determines a rule needs an update (due to new official data, a LEVEL-3 block, or a staleness flag):
1. Log the old rule version in lore-history.md under "RULE HISTORY — [RULE ID]".
2. Apply the updated rule immediately for the current and all future posts.
3. Do NOT break continuity — if the old rule produced lore already posted, honour that lore. The update applies forward only.
4. Log "RULE UPDATED — [RULE ID] — [brief description of change] — [UTC timestamp]" in lore-history.md.

### RCG-5 — Art Rule Conflict Check (image generation only)
Before generating any image, the agent runs a dedicated art-rule conflict check:
(a) Does the clothing/uniform for the featured character meet UF-0 (main uniform default) OR is a valid UF exception trigger active?
(b) Does the bonnet meet BF-1 (96% fidelity) with only BF-2 (4% creative zone) variations?
(c) Is the bonnet from the correct source (dedicated page > Layer 1+2 base > BF-3 fallback)?
(d) Does the IVES art style axis (H) conflict with the scene type or lore mood? If yes, re-roll H.
If any check fails, the agent fixes it before generating the prompt. It does NOT skip the fix and generate anyway.

### RCG-6 — Lore Flow Test
After generating lore text (before posting), the agent does a final quick self-check:
"If I read only the last 3 posts plus this new post, does the saga feel like a natural, living, evolving story? Or does it feel like a list of rules being executed?"
If the answer is "list of rules": the agent rewrites the post once, prioritising narrative voice and organic flow, while still honouring all active rules.

### RCG-7 — Conflict Log Access
The lore-history.md file maintains a running "CONFLICT LOG" section at the bottom. Every RCG action (level 1, 2, or 3 blocks, rule updates, stale flags) is appended here with UTC timestamp. This log is checked at the start of every 2-hour run before any generation begins.

### RCG-8 — Human Override Gate
Any rule in this file can be manually overridden by the user at any time by posting a new rule or instruction. The agent immediately applies the override, logs it under "HUMAN OVERRIDE — [UTC timestamp]" in lore-history.md, and treats the override as the new locked rule from that point forward.
